Implementors should periodically check the SMT32 online resources (see #Online Resources tab) for the current status of SMT32 documentation.

Abstract

The Signed Merkel Tree 32 Byte (SMT32) specification provides a cryptographically secured method by which to generate, validate and verify the integrity of a Merkel Tree & the relationship between the leaf and branch nodes in a trustless and cryptographically secure manner.

The SMT32 specification is inspired by both the patent-free SSF32 specification and SDF32 specification.

This specification is designed to provide a poratble, legally unencumbered, highly-secure, well-specified standard for encoding and decoding signed merkel trees, which must be structured as an RFC-2045 compliant Base64 encoded, 'gzdelate'd JSON file (or other compatible markup or data interchange format).

The initial motivation for the SMT32 standard was to have a convenient, extensible, compact & cryptographically secure method for storing hashes and signatures for the secure verification of ownership of files or strings transmitted across internet protocols or stored on blockchains to allow for a public data structure which may be shared and queried by many parties where the Tree's leaves may only be verifiable by the signer of the structure in a manner akin to Zero Knowledge Proofs (zk Proofs).

All applications which encode or decode SMT32 files in compliance with this specification should utilize the industry standard OpenSSL Protocol or another secure & tested cryptographic protocol.

The SMT32 file format is a patent-free format licensed under the Apache 2.0 License.

This specification defines the Internet Media Type "application/smt32" for all files while are compliant with this standard.

Please review the official repository's LICENSE.md for more information.

SMT32 features retained from the SSF32 Standard & SDF32 Standard include:

  • Complete hardware and platform independence.
  • Effective, 100% lossless compression via gzdeflate.
  • Reliable, straightforward detection of data/file corruption.

SMT32 is designed to be:

  • Simple & portable.
  • Secure: Reasonably cryptographically resistant to tampering or data corruption.
  • Legally unencumbered: to the best knowledge of the SMT32 authors, no algorithms under legal challenge are used.
  • Well compressed, implementing 'gzdeflate' algorithm.
  • Flexible: Allows for future extensions without compromising the fundementals of the specification.
  • Robust: supports full file integrity, including: simple, quick detection of common transmission errors, while also supporting highly optimized, industry standard cryptographic validation and compression algorithms.
  • Proof of Identity/Ownership: Easily prove you own your identity to the public without anyone needing to know your personal information.
  • Trustless: By publicly share your creditials where you and only you can prove you are who you attest to be, eliminating the need to share your personal information with a thrid-party.

The primary purpose of this specification is to provide the definition of the Signed Merkel Tree 32 Byte Format and recommendations for the minimum required behaviour for encoders and decoders.

Deflate/Inflate Compression

SMT32 utilizes the deflate/inflate compression method.

Deflate compression is an LZ77 derivative used in zip, gzip, pkzip, and related programs. Extensive research has been done supporting its patent-free status.

Portable C implementations are freely available across multiple platforms.

Deflate-compressed SDF32 datastreams are stored in the unstructured 'gzdeflate' format before being encoded in Base64 (to retain data integrity during transmission across non 8-bit ASCII/UFT-8 internet protocol layers).

Further details on this format are given in the zlib specification [RFC-1950].

File Name Extension

On systems where file names customarily include an extension signifying file type, the extension ".smt32" is recommended for SMT32 files.

Lowercase ".smt32" is preferred if file names are case-sensitive.

In the event an SMT file extension lacks an integer suffix (eg. ".smt"), it must be parsed as a 32 Byte SMT file as outlined in this specification.

Internet Media Type

For robustness, decoders may choose to support the interim media type: 

'application/x-smt32'
Until an official MIME type registration is complete.

Security Considerations

All SMT32 files must be non-executable & read-only.

A decoder that fails to complete all checks specified in the #Decoders tab must consider the decoded data corrupt or otherwise invalid.

Glossary of Terms

Byte
Eight bits; also called an octet.
SMT32
An secure, non-executable RFC-2045 compliant base64 encoded, compress JSON format for repressenting Signed Merkel Trees in a manner which is resistant to corruption, transferable over common internet protocols as a header, embeded within an HTML webpages, included as an email attachment or stored in a database server or blockchain.
